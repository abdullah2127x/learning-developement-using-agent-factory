# Sanity CMS + GROQ + TypeScript for Next.js App Router

## Mental Model

```
Schema (defineType/defineField)
       ↓
Sanity Studio (edits content)
       ↓
Content Lake (stores data)
       ↓
GROQ Query (fetches + shapes)
       ↓
TypeGen (generates TypeScript types)
       ↓
sanityFetch (Next.js caching)
       ↓
Server Components (renders data)
```

**Key concept**: Content is authored in Sanity Studio, queried via GROQ, and TypeGen generates types from both your schema AND your queries — so frontend types always match what you fetch.

## Quick Start (5 Steps)

### 1. Install

```bash
npm install next-sanity @sanity/image-url @portabletext/react
# TypeGen
npm install --save-dev @sanity/cli
```

### 2. Create client (sanity/lib/client.ts)

```typescript
import { createClient } from "next-sanity";

export const client = createClient({
  projectId: process.env.NEXT_PUBLIC_SANITY_PROJECT_ID!,
  dataset: process.env.NEXT_PUBLIC_SANITY_DATASET!,
  apiVersion: "2024-01-01",
  useCdn: true,
});
```

### 3. Define a schema

```typescript
// sanity/schemas/postType.ts
import { defineField, defineType } from "sanity";

export const postType = defineType({
  name: "post",
  title: "Post",
  type: "document",
  fields: [
    defineField({ name: "title", type: "string", validation: (r) => r.required() }),
    defineField({ name: "slug", type: "slug", options: { source: "title" }, validation: (r) => r.required() }),
    defineField({ name: "publishedAt", type: "datetime" }),
    defineField({ name: "image", type: "image", options: { hotspot: true } }),
    defineField({ name: "body", type: "array", of: [{ type: "block" }] }),
  ],
});
```

### 4. Write typed query

```typescript
// sanity/lib/queries.ts
import { defineQuery } from "next-sanity";

export const POSTS_QUERY = defineQuery(`
  *[_type == "post" && defined(slug.current)] | order(publishedAt desc) [0...12] {
    _id, title, slug, publishedAt
  }
`);
```

### 5. Fetch in Server Component

```typescript
// app/posts/page.tsx
import { client } from "@/sanity/lib/client";
import { POSTS_QUERY } from "@/sanity/lib/queries";

export default async function PostsPage() {
  const posts = await client.fetch(POSTS_QUERY);
  return <ul>{posts.map(p => <li key={p._id}>{p.title}</li>)}</ul>;
}
```

## Decision Trees

### Schema: document vs object?

```
Has its own URL / managed independently? → document
  └─ Embedded within another document? → object
        └─ Reused across multiple documents? → named object type
              └─ Used once inline? → inline object definition
```

### References vs Embedded Objects?

```
Content is independently managed (has own studio page)?→ reference
  └─ Content is always owned by parent? → embedded object/array
        └─ Large sets of items? → reference with GROQ join
```

### GROQ: query vs projection?

```
Need to filter documents? → *[_type == "post" && condition]
Need to shape output? → { field1, field2, "computed": expr }
Need related document data? → references->  (dereference operator)
Need reverse joins? → *[_type == "comment" && references(^._id)]
```

### TypeScript Types Strategy?

```
Have Sanity CLI? → TypeGen (recommended — auto-generates from schema + queries)
  └─ No CLI? → Manually define interfaces matching GROQ projection
```

## Production Checklist

- [ ] `apiVersion` set to a stable date string (`YYYY-MM-DD`)
- [ ] `useCdn: true` for production reads, `false` for draft mode
- [ ] All queries use `defineQuery()` for TypeGen support
- [ ] TypeGen run after schema changes (`sanity typegen generate`)
- [ ] Images always use `urlFor().width(n).auto('format').url()` (not raw `asset.url`)
- [ ] Portable Text rendered with `@portabletext/react` (not raw JSON)
- [ ] Next.js `revalidate` or `tags` set on `client.fetch()` calls
- [ ] Slug fields use `options: { source: 'title' }` for auto-generation
- [ ] Required fields validated with `validation: (r) => r.required()`
- [ ] Image fields use `options: { hotspot: true }` for smart cropping
